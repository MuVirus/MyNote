## 2、原则

# 创建型模式

## 概述
### 含义
专门管**对象怎么被创建出来**。
目标是：把 **“对象的创建”** 和 **“对象的使用”** 分开，降低耦合，让创建更灵活、更可控。

### 解决什么问题
- 直接 new 对象太死板，耦合高
- 对象创建逻辑复杂，到处重复
- 需要控制实例数量（比如只允许一个）
- 需要动态决定创建哪种对象

### 一句话精髓
**控制对象的诞生，让创建更灵活，不与业务强绑定。**

### 包含
单例、工厂方法、抽象工厂、建造者、原型。

## 1、单例
**确保一个类在整个程序生命周期中只有一个实例，并提供一个全局访问点来获取该实例**。

单例模式主要有两个目标：
1. **控制对象创建**：确保实例只被创建一次。
2. **全局可访问**：提供一个全局的访问接口。

为了实现这两个目标，一个标准的单例类通常具备以下特征：
- **私有化构造函数**：防止外部代码通过 `new` 来创建实例。
- **禁用拷贝和赋值**：通过 `delete` 关键字删除拷贝构造函数和赋值运算符，防止实例被复制。
- **提供静态访问方法**：通过一个公有的静态成员函数（如 `getInstance()`）来返回唯一的实例。

引用场景：
1. 资源管理器
	1. 数据库/连接池
	2. 线程池
	3. 缓存系统
2. 全局配置与工具类
	1. 配置管理器
	2. 日志记录器
	3. 硬件驱动访问
3. 框架与容器

## 2、工厂模式
C++ 中的工厂模式是一组用于**对象创建**的设计模式，旨在将对象的**创建过程**与**使用过程**分离，提升代码的灵活性、可维护性和可扩展性。其核心思想是：**将“new”操作符的使用抽象化、集中化**。
主要包含三种变体：简单工厂、工厂方法和抽象工厂。

- **简单工厂**：一个工厂，用 `switch`/`if` 判断创建哪种产品。新增产品要改工厂代码，违反开闭原则。
- **工厂方法**：一个工厂子类对应**一种**产品。新增产品只需加新工厂类，不改已有代码。
- **抽象工厂**：一个工厂子类对应**一整族**相关产品（如 Windows 工厂同时生产按钮、菜单、滚动条）。横向扩展产品族容易，纵向新增产品种类困难。

| 特性       | 简单工厂            | 工厂方法          | 抽象工厂                            |
| -------- | --------------- | ------------- | ------------------------------- |
| **核心目的** | 分离单个对象的创建       | 将单个对象的创建延迟到子类 | 创建**一系列（族）** 相关对象               |
| **开闭原则** | **违反**（修改工厂类）   | **符合**（扩展新工厂） | **横向符合，纵向违反**                   |
| **复杂度**  | 低               | 中             | 高                               |
| **产品数量** | 单一产品，多个变体       | 单一产品，多个变体     | **多个产品**，多个产品族                  |
| **典型比喻** | 代工厂根据订单生产不同型号手机 | 不同品牌的手机专卖店    | **生产一整套**同品牌、同风格的电子产品（手机、耳机、手表） |

# 结构性模式
### 含义
关注**类与对象如何组合、拼装成更大的结构**。
不是创建单个对象，而是**把多个对象搭成一个好用的整体**。

### 解决什么问题
- 现有接口不兼容，想让它们一起工作
- 想给对象动态加功能
- 想简化复杂系统的调用
- 想复用对象、节省内存
- 想把整体与部分统一看待（树形结构）

### 一句话精髓
**把类或对象搭积木一样组合起来，形成更灵活、更简洁的结构。**

### 包含
适配器、装饰器、代理、外观、桥接、组合、享元。
## 1、享元
享元模式（Flyweight Pattern）本质上是一种**以时间换空间**的优化策略。它的核心目的非常明确：**减少内存占用**。
### 1. 核心概念：内与外的分离
享元模式之所以能省内存，是因为它做了一个关键的动作：**状态分离**。它将对象的状态拆分为两部分：
*   **内部状态（Intrinsic State）：**
    *   **特点：** 不变的、可共享的、存储在享元对象内部的。
    *   **你的代码中：** `Font` 类中的 `name`（如“宋体”）就是内部状态。无论你在文档里用多少次“宋体”，这个字符串只需要在内存里存一份。
*   **外部状态（Extrinsic State）：**
    *   **特点：** 变化的、不可共享的、由客户端维护并在使用时传入的。
    *   **你的代码中：** 虽然你的示例比较简单，但在真实的文本编辑器场景中，**字体的大小、颜色、在屏幕上的坐标**就是外部状态。这些状态不应该存在 `Font` 对象里，否则就无法共享了。
**形象的比喻：**
想象一家咖啡馆。
*   **内部状态（享元）：** 咖啡杯、咖啡壶、桌椅。这些是店里固定的资源，大家轮流用。
*   **外部状态：** 顾客点的“拿铁”还是“美式”，顾客坐在“1号桌”还是“2号桌”。服务员（客户端）负责把咖啡（内部状态）端到指定的桌子（外部状态）给指定的客人。
### 2. 结构角色详解
一个完整的享元模式通常包含以下四个角色，我们可以看看你的代码是如何对应的，以及还可以补充什么：
ps：这里用到了工厂，所以说可以使用单例模式保证工厂只会创建一次。

| 角色                           | 职责                             | 你的代码对应          | 补充说明                                                    |
| :--------------------------- | :----------------------------- | :-------------- | :------------------------------------------------------ |
| **Flyweight (抽象享元)**         | 定义接口，声明可以接收**外部状态**的方法。        | `Font` 类        | 你的 `Font` 既是抽象也是具体。如果有 `draw(x, y)` 方法，`x, y` 就是外部状态参数。 |
| **ConcreteFlyweight (具体享元)** | 实现接口，存储**内部状态**。               | `Font` 类        | 必须保证内部状态不可变（Immutable），否则一个客户端改了，会影响其他所有使用者。            |
| **FlyweightFactory (享元工厂)**  | **维护**缓存**池**（Map），管理对象的创建与复用。 | `FontFactory` 类 | 核心逻辑是 `if (exists) return old; else create new;`。       |
| **Client (客户端)**             | **维护外部状态**，向工厂请求享元。            | `main` 函数       | 只有客户端知道“我现在需要把宋体用在第5行”。                                 |

### 3. 为什么要用享元？（优缺点分析）
**优点：**
*   **极大减少内存占用：** 这是最大的优势。如果你的系统需要显示 10,000 个字符，但只有 5 种字体样式。
    *   **不用享元：** 创建 10,000 个对象。
    *   **使用享元：** 创建 5 个享元对象 + 10,000 个轻量级上下文（只存坐标等外部状态）。
*   **提高性能：** 减少了对象创建和垃圾回收（GC）的压力。

**缺点：**
*   **系统复杂度增加：** 你需要把原本简单的对象拆得支离破碎，分离出内部和外部状态，代码逻辑会变复杂。
*   **外部状态维护成本：** 客户端必须自己维护外部状态，并且要在调用时正确传入，如果传错了，显示就会出问题。
### 4. 经典应用场景
除了你写的字体管理，享元模式在这些地方非常常见：
*   **游戏开发（最典型）：**
    *   **场景：** 一个即时战略游戏（RTS）里有 1000 个“步兵”单位。
    *   **享元：** “步兵”的模型、贴图、攻击力、移动速度是**内部状态**（所有步兵共享一份）。
    *   **外部：** 每个步兵的坐标 (x, y)、当前血量是**外部状态**。
*   **Java 的字符串常量池：**
    *   当你写 `String s1 = "abc"; String s2 = "abc";` 时，Java 实际上只创建了一个 "abc" 对象，`s1` 和 `s2` 指向同一块内存。这就是享元模式的应用。
*   **数据库连接池：**
    *   虽然通常被称为“对象池模式”，但其核心思想也是复用昂贵的连接对象（内部状态），而不是每次操作都新建连接。

# 行为型模式
## 含义
关注**对象之间的通信、职责分配、流程控制、算法交互**。
解决的是 “对象之间怎么协作、怎么传递消息、怎么分工”。

## 解决什么问题
- 多个对象互相调用，关系混乱
- 需要松耦合的消息通知（如事件）
- 需要把请求封装成对象（可撤销、排队）
- 需要按顺序处理请求
- 需要在不修改类的前提下增加新操作
- 需要管理状态切换

## 一句话精髓
**管对象之间怎么说话、怎么分工、怎么协作完成任务。**

## 包含
策略、模板方法、观察者、迭代器、责任链、命令、备忘录、状态、访问者、中介者、解释器。

## 1、策略
策略模式通常由三个角色构成：
1. **抽象策略 (Strategy)**  
    这是一个接口或抽象类，它定义了所有具体策略必须实现的公共方法。它为上下文（Context）提供了一个统一的调用标准。
2. **具体策略 (Concrete Strategy)**  
    这些是实现抽象策略接口的类。每个具体策略类都封装了一种具体的算法或行为。
3. **上下文 (Context)**  
    它持有一个指向抽象策略对象的引用（通常是指针或智能指针）。上下文不关心具体使用哪种策略，它只负责通过策略接口来调用算法。客户端可以在运行时为上下文配置或切换具体的策略。

例子（折扣计算系统）
**抽象策略类**：定义折扣计算的统一接口
**具体策略类：** 根据不同类有不同的折扣计算算法
**上下文类：** 总价+具体的折扣策略（用指针包含）

通过上下文类对象，可以切换不同的折扣策略，得到不同的最终价格。
``` cpp
int main() {
    Context context = Context(1000, std::make_unique<NormalStrategy>());
    std::cout << "最终价格: " << context.getFinalPrice() << std::endl;

    context.setStrategy(std::make_unique<VipStrategy>());
    std::cout << "最终价格: " << context.getFinalPrice() << std::endl;

    context.setStrategy(std::make_unique<SuperVipStrategy>());
    std::cout << "最终价格: " << context.getFinalPrice() << std::endl;

    return 0;
}
```
